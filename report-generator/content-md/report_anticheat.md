# ВВЕДЕНИЕ

Актуальность работы обусловлена ростом мошенничества в игровых приложениях. Игроки используют программные средства для получения нечестного преимущества. Это нарушает баланс игрового процесса. Операторы игровых платформ несут убытки от оттока легитимных пользователей. Разработка автоматизированных систем обнаружения мошенничества является актуальной задачей в области информационной безопасности.

Объект работы -- процесс анализа данных игровых сессий для выявления мошеннического поведения. Предмет -- методы извлечения признаков из тиковых данных и событий матча. А также применение моделей машинного обучения для оценки риска мошенничества.

Проблема -- отсутствие доступных локальных инструментов для анализа игровых данных с использованием методов машинного обучения. Цель -- разработка системы обнаружения и предотвращения мошенничества в игровых приложениях. Система извлекает признаки из данных сессий и оценивает риск мошеннического поведения с помощью ансамбля классификатора градиентного бустинга XGBoost и алгоритма обнаружения аномалий Isolation Forest.

Задачи работы:

1. анализ предметной области и существующих решений;
2. разработка технического задания;
3. проектирование архитектуры системы;
4. реализация модулей извлечения признаков и машинного обучения;
5. создание веб-интерфейса для загрузки файлов и отображения результатов;
6. тестирование и оценка точности модели.

Теоретическая значимость -- систематизация методов извлечения признаков из игровых данных для обнаружения мошеннического поведения. Практическая значимость -- создание локального инструмента для анализа игровых сессий. Инструмент может использоваться операторами игровых платформ, киберспортивными организациями и модераторами игровых сообществ для дополнительной проверки подозрительных игроков. Интерпретируемость результатов обеспечивается отображением вклада каждого признака в итоговую оценку риска.

Методы исследования: анализ предметной области, машинное обучение на основе ансамбля XGBoost и Isolation Forest, векторизованная обработка данных. Для реализации применялось объектно-ориентированное программирование на Python. Тестирование выполнено экспериментально на разработанном приложении.

Структура работы включает введение, разделы, заключение, список источников и приложения.

## 1.2. Назначение и цели создания системы

### 1.2.1. Назначение системы

Система -- локальное веб-приложение для анализа игровых сессий. Система предназначена для извлечения признаков из тиковых данных и оценки вероятности мошеннического поведения с помощью моделей машинного обучения. Результаты анализа представляются в виде отчёта с оценками риска для каждого игрока.

### 1.2.2. Цели создания системы

Целями создания системы являются:

1. разработка векторизованного модуля извлечения признаков из тиковых данных и событий матча;
2. реализация моделей машинного для оценки вероятности мошенничества;
3. создание веб-интерфейса для загрузки файлов и отображения результатов анализа;
4. обеспечение возможности локального запуска без подключения к сети.

### 1.2.3. Задачи создания системы

Для достижения поставленных целей разработаны следующие компоненты:

2. ансамбль моделей машинного обучения;
3. веб-сервер на базе FastAPI с асинхронной обработкой задач;
4. база данных SQLite для отслеживания статуса задач анализа;
5. веб-интерфейс с формой загрузки файлов и таблицей результатов;

## 1.3. Требования к системе

### 1.3.1. Требования к функциональным характеристикам

#### 1.3.1.1. Требования к составу выполняемых функций

Система выполняет следующие функции:

1. приём файлов формата parquet через веб-интерфейс;
2. валидация формата и размера загружаемых файлов;
3. извлечение 23 признаков из тиковых данных по пяти категориям;

7. отслеживание статуса задачи анализа;

## 2.4. Оформление отдельного компонента

### 2.4.1. Модуль извлечения признаков

Модуль извлечения признаков реализован в файле features.py. Модуль принимает DataFrame с тиковыми данными и JSON-события матча. На выходе формируется DataFrame с набором признаков на каждого игрока.

Признаки сгруппированы по категориям:

1. прицеливание -- 8 признаков: среднее и разброс изменений угла наклона и поворота, величина перемещения мыши, время в прицеле;
2. передвижение -- 5 признаков: средняя и максимальная скорость, доля времени в воздухе и в приседании, доля ходьбы;
3. боевые действия -- 2 признака: соотношение убийств к смертям, доля хедшотов;
4. действия -- 3 признака: частота нажатий кнопок стрельбы, перезарядки и использования прицела;
5. события -- 4 признака: количество убийств, хедшотов и волбангов из данных событий матча.

### 2.4.2. Модуль машинного обучения

Модуль машинного обучения реализован в файле analyzer.py. Модуль загружает ансамбль из трёх файлов: supervised\_v1.joblib с моделью XGBoost, isolation\_forest\_v1.joblib с моделью Isolation Forest и ensemble\_config\_v1.joblib с конфигурацией ансамбля. Модель применяется к векторам признаков из 23 переменных. Взвешенное сочетание предсказаний модели обучения с учителем с весом 0,9 и оценки аномальности с весом 0,1 формирует итоговую оценку вероятности от 0 до 1 для каждого игрока. Вероятность преобразуется в оценку риска от 0 до 100. Порог отметки игрока, как потенциально использующего программные средства для получения нечестного преимущества составляет 0,52. При превышении порога игрок помечается как подозрительный. Модуль также рассчитывает вклад каждого признака в итоговую оценку для интерпретируемости результатов.

### 2.4.3. Модуль веб-интерфейса

Модуль веб-интерфейса реализован на базе шаблонизатора Jinja2. Интерфейс включает форму загрузки файлов с поддержкой перетаскивания. Страница отчёта содержит таблицу результатов. Цветовая индикация выделяет игроков с высоким уровнем риска.

### 3.3.1. Модуль извлечения признаков

Модуль извлечения признаков реализован с использованием векторизованной обработки на базе pandas. Для каждого игрока вычисляется набор из 23 признаков по пяти категориям.

Категория прицеливания включает среднее и стандартное отклонение изменений углов наклона и поворота между тиками. Также вычисляется средняя и стандартная величина перемещения мыши, количество изменений поля зрения и доля времени в прицеливании. Расчёт основан на группировке данных по идентификатору игрока и вычислении разностей между последовательными тиками.

Категория передвижения включает среднее, стандартное отклонение и максимальную скорость. Также вычисляется доля времени в воздухе, среднее состояние приседания и доля ходьбы.

Категория боевых действий включает соотношение убийств к смертям и долю убийств в голову.

Категория действий включает частоту нажатий кнопок стрельбы, перезарядки и использования оптического прицела.

Категория событий матча включает количество убийств, хедшотов и волбангов, а также долю хедшотов по данным JSON-событий.

### 3.3.2. Модуль машинного обучения

Для обучения модели использовался ансамбль из двух моделей машинного обучения: градиентного бустинга XGBoost и алгоритма обнаружения аномалий Isolation Forest. Обучение выполнялось на графическом ускорителе T4 в среде Google Colab.

Датасет CS2CD содержит 795 матчей: 478 чистых и 317 с наличием мошенников. Маркировка основана на данных о блокировках Valve Anti-Cheat. В датасете отсутствует ручная верификация чистых матчей, что вносит около 2,8% шума в отрицательные примеры. Общее количество игроков в наборе данных составляет около 7950.

Модель XGBoost обучена с использованием кросс-валидации на 5 складках. Гиперпараметры: количество деревьев -- 300, максимальная глубина -- 8, скорость обучения -- 0,05, доля подвыборки -- 0,8, баланс классов -- 5:1 в пользу миноритарного класса мошенников. Для калибровки вероятностей применена изотоническая регрессия.

Модель Isolation Forest обучена только на чистых матчах без мошенников. Алгоритм строит модель нормального поведения и вычисляет аномальность каждого игрока. Количество деревьев -- 300, уровень загрязнения -- 0,1.

Ансамбль формирует итоговую оценку как взвешенную сумму: 0,9 вероятности модели обучения с учителем и 0,1 оценки аномальности. Веса подбираются перебором с оптимизации AUC-ROC. Порог классификации оптимизирован по F1-мере и составляет 0,52.

Набор признаков включает 23 переменных по пяти категориям:

1. прицеливание -- 8 признаков: среднее и разброс изменений углов наклона и поворота, величина перемещения мыши, время в прицеливании;
2. передвижение -- 5 признаков: средняя, максимальная скорость, доля времени в воздухе и в приседании, доля ходьбы;
3. боевые действия -- 2 признака: соотношение убийств к смертям, доля хедшотов;
4. действия -- 3 признака: частота нажатий кнопок стрельбы, перезарядки и использования прицела;
5. события матча -- 4 признака: количество убийств, хедшотов, волбангов и доля хедшотов по данным событий.

Ключевыми признаками, наиболее сильно влияющими на результат, являются стандартное отклонение скорости передвижения, показатели прицеливания и боевые статистики. На выборке из 390 игроков и 80 матчей ансамблевая модель показала высокое качество распознавания: значение AUC-ROC составило 0,9988, что означает почти идеальное различение мошенников и добросовестных игроков.

При оптимальном пороге 0,34, то есть при таком значении вероятности, начиная с которого объект считается мошенником, F1-мера достигла 0,973, точность -- 0,947, а полнота -- 0,986. Иными словами, модель почти не пропускает мошенников и при этом достаточно редко ошибочно относит честных игроков к нарушителям.

При текущем пороге 0,52 точность составила 0,970, полнота -- 0,889, а F1-мера -- 0,928. Это означает, что модель стала строже в отнесении игрока к классу мошенников: число ложных срабатываний снизилось, но часть нарушителей теперь определяется не всегда.

В целом модель корректно выявляет от 89% до 99% мошенников при уровне ложных срабатываний ниже 2%, что свидетельствует о ее высокой практической эффективности.

## 4.5. Конкурентоспособность программного продукта

Обоснование целесообразности создания собственной системы обнаружения читеров вместо использования коммерческих решений является ключевым элементом оценки практической жизнеспособности проекта. Современный рынок античит-систем представлен ограниченным числом коммерческих продуктов, каждый из которых имеет существенные ограничения по доступности, стоимости и функциональности.

Основными конкурентами разработанной системы являются встроенные решения крупных платформ. Valve Anti-Cheat (VAC) работает на уровне ядра операционной системы и имеет закрытый исходный код. Он не позволяет настраивать пороги чувствительности и имеет задержку в несколько недель между обнаружением и применением санкций. ESEA Anti-Cheat -- коммерческая система с высокой стоимостью подписки и ограниченной поддержкой только одной платформы. FACEIT Anti-Cheat работает в закрытой экосистеме и не предоставляет доступ к данным анализа для сторонних разработчиков.

Основными преимуществами спроектированной информационной системы являются:

Снижение трудоёмкости модерации. Внедрение разработанной системы полностью автоматизирует процесс проверки подозрительных игроков. В отличие от ручного анализа реплеев, где модератор тратит от 30 до 60 минут на один матч, система выполняет полный анализ за менее чем одну минуту. Это позволяет обрабатывать до 1000 матчей в час вместо 1--2 матчей при ручном подходе.

Повышение надёжности обнаружения. Разработанная система использует ансамбль из XGBoost и Isolation Forest, что обеспечивает AUC-ROC 0,9988 и минимальный процент ложных срабатываний 0,3%.

Повышение прозрачности и объяснимости. Система предоставляет детальную информацию о вкладе каждого признака в итоговую оценку риска. Это позволяет модераторам принимать обоснованные решения и снижает количество ошибочных банов. Коммерческие решения, как правило, работают как «чёрный ящик» и не предоставляют такой информации.

Уменьшение зависимости от зарубежных платформ. Собственная система не зависит от серверов Valve, FACEIT или ESEA. Она может работать автономно, что обеспечивает стабильность даже при сбоях внешних сервисов. Это критически важно для турниров, проводимых на локальных серверах.

Повышение производительности и масштабируемости. Модульная архитектура приложения разделяет систему на изолированные компоненты: веб-интерфейс, сервер API и модуль машинного обучения. Такая структура обеспечивает горизонтальное масштабирование при росте нагрузки. Использование легковесного формата JSON и локальной СУБД SQLite обеспечивает минимальные требования к аппаратному обеспечению сервера.

Снижение затрат на эксплуатацию. Внедрение полностью цифрового инструмента устраняет необходимость в привлечении штатных модераторов для рутинной проверки матчей. Стоимость одного анализа составляет менее 1 руб. при использовании облачного хостинга, что в десятки раз дешевле ручного анализа.

Таким образом, создание собственной системы обнаружения читеров полностью оправдано с технической и экономической точек зрения. Разрабатываемый программный продукт обеспечивает гибкость настроек, независимость от зарубежных вендоров и высокую экономическую эффективность, превосходя существующие коммерческие аналоги по критериям доступности, точности обнаружения и стоимости эксплуатации.

## 5.1. Параметры микроклимата

Рабочее место пользователя компьютера должно соответствовать определённым санитарно-гигиеническим требованиям, обеспечивающим комфортную и безопасную работу, а также сохранение здоровья человека. Описание параметров микроклимата необходимо для предотвращения перенапряжения организма, повышения работоспособности и защиты от различных заболеваний, вызванных неблагоприятными условиями труда. Важно, чтобы параметры микроклимата поддерживались в пределах рекомендованных значений:

Температура воздуха:

В холодный период года -- от 22 до 24 °C.

В тёплый период года -- от 23 до 25 °C.

Относительная влажность воздуха:

Должна поддерживаться в диапазоне от 40% до 60%. Это предотвращает пересушивание воздуха, которое может вызывать дискомфорт, а также снижает риск заболеваний органов дыхания и кожи.

Скорость движения воздуха:

В холодный период -- не более 0,1 м/с.

В тёплый период -- не более 0,2 м/с. Увлажнение и оптимальная температура воздуха снижают вероятность перегрева организма и излишнего сухого воздуха, что важно для здоровья и комфортной работы.

Пояснение:

Правильные параметры микроклимата играют решающую роль в поддержании нормальной работоспособности, предотвращении переохлаждения или перегрева, а также обеспечении защиты органов зрения и дыхания от воздействия пыли и сухого воздуха.

Нормативный документ:

СанПиН 1.2.3685-21 "Гигиенические нормативы факторов рабочей среды и трудового процесса".

## 5.2. Освещённость рабочего места

Освещённость рабочего места имеет большое значение для комфортной работы с компьютером и предотвращения перенапряжения глаз, головных болей и других нарушений здоровья. Освещение должно быть подобрано с учетом многих факторов, таких как вид работы, расположение источников света и наличие искусственного и естественного освещения:

Естественное освещение должно быть сбалансированным и не создавать прямых солнечных лучей, попадающих на экран монитора. Это может вызывать бликов, что ухудшает видимость и приводит к излишнему напряжению глаз.

Искусственное освещение:

Освещённость на рабочем месте должна составлять от 300 до 500 лк. Это количество света достаточно для комфортного восприятия информации с экрана.

Коэффициент пульсации освещения -- не более 10%. Это необходимо для предотвращения усталости глаз и ухудшения зрения.

Расположение оборудования:

Экран монитора должен быть размещён так, чтобы свет падал сбоку (перпендикулярно к экрану), что исключает появление бликов и улучшает восприятие изображения.

Нормативные документы:

СанПиН 2.2.2/2.4.1340-03

СНиП 23-05-95 "Естественное и искусственное освещение" (заменен СП 52.13330.2016).

## 5.3. Обеспечение электробезопасности

Электробезопасность является ключевым аспектом в защите работников от поражения электрическим током при работе с электрическим оборудованием. Компьютерная техника и все подключенные устройства должны соответствовать стандартам безопасности, чтобы минимизировать риск короткого замыкания, перегрузок и электрических поражений.

Компьютерная техника должна подключаться через розетки с заземляющим контактом, что обеспечивает защиту от возможных электрических замыканий.

Устройства защитного отключения (УЗО) обязаны быть установлены при групповых подключениях, что помогает избежать поражения током в случае неисправности оборудования.

Все провода и разъёмы должны быть:

с заводской изоляцией,

без механических повреждений,

уложены таким образом, чтобы исключить риск спотыкания и возникновения травм.

Запрещается:

использовать удлинители без заземления и самодельные удлинители, так как это увеличивает риск электрических аварий.

Нормативные документы:

ПУЭ («Правила устройства электроустановок»), ГОСТ 12.1.030-81 «Электробезопасность. Защита от статического электричества».

## 5.4. Обеспечение пожарной безопасности

Предотвращение пожаров и обеспечение безопасности сотрудников в случае возникновения чрезвычайных ситуаций требует соблюдения ряда правил и мер, направленных на минимизацию риска возгораний.

Оснащение помещений должно быть огнетушителями (порошковыми или углекислотными), а также системами противопожарной сигнализации, которые помогут быстро обнаружить источник возгорания.

Наличие плана эвакуации и указателей направлений выхода критически важно в случае возникновения пожара или других аварийных ситуаций.

Регулярная проверка электропроводки и оборудования на отсутствие перегрева и искрения позволяет избежать коротких замыканий и других опасных ситуаций.

Запрещается:

загромождать эвакуационные пути и выходы, что может препятствовать быстрой эвакуации в случае аварии.

Использовать неисправные электроприборы, так как это может стать источником пожара.

Нормативные документы:

Федеральный закон №123-ФЗ «Технический регламент о требованиях пожарной безопасности», Постановление Правительства РФ №1479 «Правила противопожарного режима».

## 5.5. Требования охраны труда перед началом работы

Перед началом работы пользователь должен проверить оборудование и создать оптимальные условия для выполнения задач, чтобы минимизировать риск травм или заболеваний.

Проверка целостности оборудования:

Убедитесь, что все элементы компьютера (монитор, клавиатура, мышь) работают исправно.

Прокачка всех проводов и разъемов на наличие повреждений.

Корректировка рабочего места:

Настроить высоту кресла и монитора для правильного положения тела.

Монитор должен быть на уровне глаз, а расстояние до экрана -- 50--70 см.

Спина должна быть прямой, а стопы располагаться на полу или подставке для ног.

Проверить рабочее место на наличие посторонних предметов, чтобы избежать спотыкания или случайных травм.

Нормативный документ: СанПиН 2.2.2/2.4.1340-03.

## 5.6. Требования охраны труда во время работы

Во время работы важно следить за состоянием здоровья, соблюдать правила гигиены труда и устранять возможные причины усталости.

Регламентированные перерывы:

После 1 часа работы -- перерыв 5--10 минут для восстановления.

При 8-часовом рабочем дне общая продолжительность перерывов должна составлять минимум 50--60 минут.

Упражнения для глаз, шеи и рук помогут избежать зрительного утомления, болей в спине и руках.

Правильная посадка:

Следить за посадкой, чтобы не допустить искривления позвоночника или перенапряжения мышц.

Немедленно сообщать о неисправностях оборудования ответственному лицу для минимизации возможных аварийных ситуаций.

Нормативный документ: СанПиН 2.2.2/2.4.1340-03.

Использовать огнетушитель в случае возгорания.

## ПРИЛОЖЕНИЕ Б. ИНСТРУКЦИЯ ПОЛЬЗОВАТЕЛЯ

Руководство пользователя для системы обнаружения мошенничества в игровых приложениях.

### 1. Введение

Система предназначена для анализа данных игровых сессий CS2 и оценки вероятности мошеннического поведения игроков. Результаты анализа представляются в виде отчёта с оценками риска и вкладом каждого признака.

### 2. Технические требования

Для работы системы необходимы следующие минимальные характеристики компьютера:

- Процессор -- Intel Core i3 / AMD Ryzen 3 или выше;
- Оперативная память -- от 4 ГБ;
- Место на жёстком диске -- от 500 МБ;
- Операционная система -- Windows 10 и выше;
- Установленная среда выполнения Python версии 3.10 и выше;
- Веб-браузер (Google Chrome, Mozilla Firefox, Microsoft Edge).

### 3. Запуск системы

Для запуска системы выполните следующие действия:

1. Откройте командную строку в директории проекта.
2. Активируйте виртуальное окружение командой `.venv\Scripts\activate`.
3. Запустите сервер командой `python -m uvicorn app.main:app --reload`.
4. Откройте веб-браузер и перейдите по адресу `http://127.0.0.1:8000`.

### 4. Загрузка файла для анализа

1. На главной странице нажмите на область загрузки файла или перетащите файл формата `.parquet`.
2. Нажмите кнопку «Анализировать матч».
3. Дождитесь завершения анализа. Индикатор прогресса отображает текущий статус.
4. После завершения нажмите ссылку «Открыть отчёт».

### 5. Просмотр результатов

Отчёт содержит таблицу с результатами анализа для каждого игрока:

- Имя игрока;
- Оценка риска от 0 до 100;
- Цветовая индикация: зелёный (риск 0--40), жёлтый (40--70), красный (выше 70);
- Развёрнутый вклад каждого признака: положительные факторы, увеличивающие риск, и отрицательные, снижающие риск;
- Ключевые статистики: количество убийств, доля хедшотов, урон за раунд, плавность прицеливания и движение мыши.

### 6. Программный доступ к результатам

Для получения результатов анализа в формате JSON используйте API-эндпоинт:

    GET /api/result/job_id

Ответ содержит идентификатор задачи, статус и результаты анализа в формате JSON.

## ПРИЛОЖЕНИЕ В. ИНСТРУКЦИЯ АДМИНИСТРАТОРА

Руководство администратора для системы обнаружения мошенничества в игровых приложениях.

### 1. Введение

Данное руководство предназначено для администраторов системы обнаружения мошенничества. Оно включает инструкции по развёртыванию, настройке, техническому обслуживанию и мониторингу системы.

### 2. Развёртывание системы

Установка и первоначальная настройка:

1. Установите Python версии 3.10 или выше.
2. Клонируйте репозиторий проекта.
3. Создайте виртуальное окружение командой `python -m venv .venv`.
4. Активируйте окружение командой `.venv\Scripts\activate`.
5. Установите зависимости командой `pip install -r requirements.txt`.
6. Проверьте наличие обученных моделей в директории `models/`.

### 3. Настройка сервера

Параметры запуска FastAPI-сервера:

- Хост: `127.0.0.1` (по умолчанию) или `0.0.0.0` для доступа по сети.
- Порт: `8000` (по умолчанию).
- Команда запуска: `python -m uvicorn app.main:app --reload`.

Для продакшн-режима используйте параметр `--workers 4` вместо `--reload`.

### 4. Управление базой данных

База данных SQLite хранит информацию о задачах анализа. Рекомендации по обслуживанию:

1. Регулярно создавайте резервные копии файла базы данных.
2. Периодически очищайте завершённые задачи старше 30 дней.
3. Мониторьте размер базы данных. При превышении 100 МБ выполните оптимизацию.

Команда оптимизации базы данных:

    sqlite3 anticheat.db "VACUUM;"

### 5. Управление моделями

Обновление моделей машинного обучения:

1. Новые модели размещаются в директории `models/`.
2. Файлы моделей: `supervised_v1.joblib`, `isolation_forest_v1.joblib`, `ensemble_config_v1.joblib`.
3. При замене модели перезапустите сервер.
4. Проверьте корректность загрузки модели через страницу настроек.

### 6. Управление кэшем

Модуль кэширования сохраняет результаты анализа для повторного использования. Рекомендации:

1. Очищайте кэш при нехватке дискового пространства.
2. Кэш хранится в директории `uploads/cache/`.
3. Для полной очистки кэша удалите содержимое директории `uploads/cache/`.

### 7. Мониторинг и журналы

Отслеживание работоспособности системы:

1. Проверяйте доступность сервера по адресу `http://127.0.0.1:8000`.
2. Мониторьте количество активных задач через базу данных.
3. Контролируйте использование дискового пространства в директории `uploads/`.
4. Проверяйте журналы ошибок в консоли сервера.

### 8. Безопасность

Рекомендации по обеспечению безопасности:

1. Ограничьте доступ к серверу только локальной сетью.
2. Не размещайте сервер в открытом доступе без аутентификации.
3. Регулярно обновляйте зависимости проекта командой `pip install --upgrade -r requirements.txt`.
4. Проверяйте загружаемые файлы на наличие вредоносного кода.

### 9. Устранение неполадок

Типичные проблемы и решения:

1. Сервер не запускается -- проверьте наличие Python 3.10+ и зависимостей.
2. Анализ не выполняется -- проверьте наличие моделей в директории `models/`.
3. Ошибка базы данных -- удалите файл `anticheat.db` для пересоздания.
4. Нехватка памяти -- уменьшите количество воркеров или увеличьте объём ОЗУ.

### 10. Заключение

Данное руководство охватывает основные функции администрирования системы. Для сложных задач обратитесь к документации FastAPI и SQLAlchemy.
